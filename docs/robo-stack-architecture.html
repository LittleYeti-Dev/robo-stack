1. Executive Summary

What is Robo Stack?

Robo Stack is a hybrid AI development stack — a fully automated, security-hardened platform that provides AI-augmented software development capabilities on AWS infrastructure managed through GitOps principles. It combines a cloud-hosted Kubernetes cluster, a Claude API proxy for AI integration, browser-based development tools, and a comprehensive CI/CD pipeline into a single, reproducible environment.

Architecture Philosophy

Hybrid Cloud

Single AWS EC2 node running K3s with cloud AI services (Claude API). Local compute for workloads, cloud for intelligence.

GitOps

GitHub Actions drives all deployments. Infrastructure is Terraform-managed. All state is version-controlled. No manual cluster operations.

Security-First

Defense in depth: code scanning (CodeQL), secret detection (gitleaks), container scanning (Trivy), SBOM generation, branch protection, RBAC at every layer.

AI-Augmented

Claude API proxy with rate limiting and metrics. AI code review in CI. Cloud-first model serving (EV2.1 decision: Claude API over local Ollama).

3. Kubernetes Layer

K3s Cluster Architecture

Robo Stack uses K3s (lightweight Kubernetes) on a single EC2 node, selected per evaluation gate EV1.1 over Minikube. K3s provides a production-grade Kubernetes API with lower resource overhead (~512 MB RAM vs. Minikube's ~1.1 GB), built-in Traefik ingress, and a single-binary deployment model ideal for the single-operator constraint.

4. Application Layer

Claude API Proxy

AI FastAPI-based reverse proxy for the Anthropic Claude API (S2.3 deliverable).

  • Runtime: Python 3.11 + Uvicorn
  • Rate Limiting: 10 requests/minute per client IP (configurable via ConfigMap)
  • Retry Logic: Exponential backoff, max 3 retries on transient failures
  • Endpoints:
    • POST /v1/messages — Proxy to Anthropic API with validation
    • GET /health — Liveness/readiness probe target
    • GET /metrics — Request count, token usage, error rates
  • Security: API key injected via K8s Secret (claude-api-credentials), CORS enabled
  • Probes: Liveness (30s delay, 30s period), Readiness (20s delay, 10s period)

Development Tools

Code Server :8443

Browser-based VS Code instance. Full IDE with terminal, extensions, and Git integration. Accessible remotely over HTTPS.

Jupyter Notebook :8888

Data science and prototyping environment. Python kernel with access to cluster services for experimentation.

Node Dev :3000

Node.js 20 development container. Hot-reload capable for frontend/API development.

Service Connectivity

All services communicate within the robo-stack namespace via ClusterIP services. Traefik (K3s built-in) handles ingress routing. The Claude Proxy is the only service that makes external API calls (to api.anthropic.com).

11. Constraints & Known Limitations

Single-Node Constraint

All workloads run on one t3.xlarge (4 vCPU, 16 GB). No HA, no pod rescheduling on failure. Node failure = full outage. Ceiling is ~75% headroom today.

No GPU

t3.xlarge has no GPU. Local model inference is not viable. All AI workloads depend on the Claude API (cloud). This is a cost vs. latency trade-off accepted in EV2.1.

Manual Secret Rotation

Secrets are stored in GitHub Secrets and K8s Secrets. No automated rotation. External Secrets Operator + Vault is planned but not deployed.

Single-Developer Model

Architecture assumes a sole operator (@LittleYeti-Dev). No on-call rotation, no team-based RBAC, no multi-tenant isolation. Branch protection is meaningful but review gates are self-approved.

No Network Policies

Cross-namespace traffic in K8s is unrestricted. Network policies are planned but not implemented. This is a gap for the prod namespace isolation requirement.

Open K8s API / kubectl Proxy

Ports 6443 and 8001 are open to 0.0.0.0/0 in the security group. Authentication is required but exposure increases attack surface. CIDR restriction recommended.
