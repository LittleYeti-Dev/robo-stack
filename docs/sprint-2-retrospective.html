Key Decisions Made

EV2.1: Cloud-First Model Serving — Approved by Yeti

All AI inference routes through Claude API (cloud). Local model serving (Ollama on EC2) rejected: 50x slower on CPU-only t3.xlarge, lower quality models, RAM contention with K3s workloads, and offline capability irrelevant on a cloud instance. Re-evaluate in Sprint 4 or when hardware changes.

GHCR over ECR for Container Registry

GitHub Container Registry (GHCR) chosen over AWS ECR for Sprint 2. Simpler auth (GITHUB_TOKEN), no additional Terraform resources, free for public repos. ECR remains available via the IAM policies if needed later for private images.

NodePort over Ingress for Dev Tools

Dev tools use NodePort services (30080, 30088) instead of K3s Traefik ingress. Simpler for single-node dev clusters, accessed via SSH tunnel. Ingress can be added in Sprint 3 if multi-service routing is needed.

Lessons Learned

1. Sprint 1 lessons actually work when enforced

Every Sprint 1 lesson was codified into standards and applied as story acceptance criteria. Result: 100% completion, zero incidents, zero post-merge CI failures. Process discipline compounds — the initial investment in S1.1 (process design) continues to pay dividends.

2. Feature branches for workflows are non-negotiable

PR #27 included 2 new GitHub Actions workflows. Both were validated in the feature branch before merge. In Sprint 1, this would have caused 2+ fix cycles on main. The 30-second cost of creating a branch saves hours of debugging.

3. Graceful degradation is a design principle

The AI code review workflow posts "Review unavailable" when the API key isn't set — it never blocks a PR merge. The Docker build skips when no Dockerfiles change. Design for failure and the system stays robust even when partially configured.

4. Cloud-first simplifies everything on cloud infrastructure

EV2.1 confirmed what was already apparent: running local models on a CPU-only cloud instance is worse in every dimension. Leaning into cloud APIs instead of fighting hardware constraints let us ship faster and with better results.

5. Velocity scales with process maturity

Sprint 1: 7/8 stories, 10 incidents, 6 CI fixes. Sprint 2: 8/8 stories, 0 incidents, 0 CI fixes. Same team, same tools, better process. The compounding effect of each sprint's retrospective feeding into the next sprint's standards is measurable.
