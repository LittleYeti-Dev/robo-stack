What Went Well

  • Full AWS infrastructure deployed via Terraform — 26 resources (VPC, EC2, IAM, security groups) provisioned from code in a single session. All IaC, no manual setup.
  • CI pipeline end-to-end operational — 5 GitHub Actions workflows (CI, CodeQL, security scan, markdown lint, local deploy) all passing green after debugging 6 issues live.
  • Process-first design paid off — S1.1 established the sprint structure, naming conventions, and agent prompts before any build work began. Zero confusion on scope or approach.
  • Claude Code/Cowork proved highly effective — handled Terraform, CI/CD debugging, scripting, documentation, issue tracking, and sprint management all within a single session.
  • Issue tracking discipline maintained — all 10 incidents logged to both the Notion Issue Registry and the GitHub wiki with root cause analysis and preventive actions.
  • Eval gates worked as intended — EV1.1 (K3s vs Minikube) produced a clear decision with documented rationale. S1.6 (Copilot test) provided actionable data for agent routing.

What Didn't Go Well

  • Copilot Agent Mode failed to deliver — scored 1/5 on the S1.6 hands-on test. Created a branch and PR but produced 0 lines of code before being cancelled at 2m41s. Not viable for agentic tasks.
  • 6 CI workflow issues discovered post-merge — ShellCheck severity, yamllint path filtering, CodeQL v2 deprecation, Node.js 16 EOL, and others required multiple fix commits. Would have been caught with a staging branch.
  • Terraform provider quirks — default_tags with timestamp() causes perpetual plan drift (known AWS provider bug). Egress rules with ip_protocol="-1" silently reject port specifications. Both required debugging time.
  • AWS credential expiration — root access keys expired mid-session. Required manual console intervention to start the EC2 instance. Need a more robust IAM setup.
  • TP1 touchpoint not completed — deferred to Sprint 2 as TP2.0. Sprint velocity was focused on build work, leaving the stakeholder review for later.

Lessons Learned

1. Process-first design eliminates ambiguity

Starting Sprint 1 with S1.1 (process design) before any build stories meant every team member (human and AI) had clear scope, naming, and acceptance criteria. Zero rework from misaligned expectations.

2. Test your CI in a branch, not on main

Six CI issues post-merge cost multiple fix-then-check cycles. A staging branch with a test run would have caught all of them before the commit history grew. Enforce this for Sprint 2.

3. Agent selection matters more than agent hype

Copilot Agent Mode scored 1/5 — it created a PR but wrote zero code. Claude handled every substantive task. Agent routing should be based on measured performance, not marketing. Always eval before you trust.

4. Document incidents as they happen

Logging all 10 issues to Notion with root cause, fix, and preventive action while the context was fresh means future sprints have a searchable knowledge base. Don't defer documentation — it decays fast.

5. Terraform provider quirks require defensive coding

The default_tags + timestamp() bug and the egress port rejection are both known issues but poorly documented. Always test terraform plan idempotency before deploying, and avoid dynamic values in provider-level config.
